# Waypoint -- Domain Context

Waypoint is a collaborative trip planning PWA. One trip owner creates a trip, invites members, and the group builds an itinerary together. After the trip, it can be closed out, archived publicly, and exported.

## Glossary

Terms are authoritative. Use these in code, issues, and conversation. Synonyms in parentheses are explicitly avoided.

| Term | Definition | Avoid |
|------|-----------|-------|
| **Trip** | Top-level aggregate. Has a slug, dates, members, and a **derived lifecycle** ŌĆö `planning` (today < start) / `active` (in range, = `isTripActive`) / `wrap-up` (today > end & not archived) / `closed` (archived). There is **no stored `status` field**; the only persisted lifecycle flag is `archived` (bool). *(D6 fix, #116: previously mislabeled "status (planning/active/closed)" as a stored 3-state field ŌĆö it is a 4-state value derived from dates + `archived`. Wrap-up/closed surfaces specced in PRD #195 (`docs/TRIP_WRAPUP_PRD.md`).)* | journey, vacation, itinerary, ~~trip status field~~ |
| **Phase** | A named time span within a trip (e.g. "Paris leg"). A trip has 0..n phases. Phases group days and items by location or theme. No individual colors ŌĆö phase indicators use the current mode's accent color. | leg, segment, section, phase color, phase palette |
| **Day** | A calendar date within a trip. Belongs to 1..n phases (multi-relation). Planned and done items are assigned to a day; unplanned and considered items may or may not have one. | date (ambiguous with datetime) |
| **Item** | A plan entry within a phase: lodging, transportation, flight, activity, meal, note, or checklist. Has a type, optional anchor times, and a status (unplanned ŌåÆ planned ŌåÆ done/considered). An unplanned item belongs to a phase and is available for any day in that phase. A planned item is assigned to a specific day. May span multiple days via end date. | event, plan, entry, booking |
| **Item Type** | The category of an item. lodging (overnight stay), transportation (movement between places ŌĆö can bridge phases), flight (air travel, has AeroDataBox integration), activity (things to do), meal (dining), note (free text). *(The former `checklist` type is superseded by the standalone Checklist primitive ŌĆö see below. Closes #45.)* | category |
| **Checklist** | A named container of Tasks with a **parent scope**: attaches to exactly one of an Item, a Phase, or a Trip. A Trip may own several (e.g. "Things to Book", "Packing List"). Not an Item ŌĆö has no votes, anchor times, or Day relation. Replaces the former `checklist` Item Type and its `checklist_items` collection. **Surfaced as 'List' / 'Lists'** in nav and UI copy (sub-tab, headings, buttons) ŌĆö the glossary noun never appears in product (#116 audit; sanctioned, mirrors [[Vote]]ŌåÆ'Pass'). | checklist item type, sub-items |
| **Task** | A single checkable line within a Checklist. Fields: title, checked (bool), optional **assignee** (one Trip Member). Not an Item. | checklist item, todo, sub-item |
| **Smart List** | A list whose rows are **projected from [[Item]]s**, not hand-entered ŌĆö in two kinds. **Checkable:** the **booking** Smart List shows every planned Item that requires booking but isn't booked, and checking a row sets the Item's `booked` flag (read-and-check-off only; projected check rows carry no own assignee or notes). **Read-only:** the **flights** Smart List, a trip-wide chronological lineup of every flight Item with its route, times, and [[Assignment\|assignee]] avatars ŌĆö a lens, nothing to check off. *(flights Smart List planned ŌĆö #210, `docs/ITEM_ASSIGNMENT_PRD.md`.)* | auto-list, generated checklist |
| **Booking Readiness** | An orthogonal binary `booked` flag on bookable Items ŌĆö *not* a Status step. An Item "requires booking" via a per-Item flag, pre-filled true for lodging/flight/transportation. The **"Needs booking"** card pill is the projection `needsBooking()` = `planned && requires_booking && !booked` (the same predicate as the booking [[Smart List]]) ŌĆö so it appears only on committed items, never on Parking Lot ideas, and is mutually exclusive with the `Booked` pill. The Tri-State Booking Pill is a separate, deferred concept. | booked status |
| **Anchor Time** | An optional start time and/or end time on an item (e.g. "09:15", "15:30"). Items without anchor times are untimed. | start time, scheduled time |
| **Time Slot** | A soft display grouping derived from anchor times: Morning (before noon), Afternoon (noonŌĆō5pm), Evening (after 5pm). Never stored ŌĆö always computed. Items without anchor times appear in an "Anytime" group. Replaces the former rigid Slot enum. | time block, period, slot (as stored enum) |
| **Sort Order** | An integer on an item used for drag-to-reorder positioning of untimed items within a day view, relative to anchored items and time slot headers. Implementation detail, not a user-facing concept. | rank, priority, order |
| **Multi-day Item** | An item with an end date that differs from its day's date. Spans multiple days (e.g. a hotel stay). Displayed at the top of each spanned day. | spanning item |
| **Ongoing** | The temporal state of an item during Trip Mode when the current time falls between its start and end. A multi-day item becomes ongoing when its start date/time is reached. A single-day item with anchor times can also be ongoing. Not a type ŌĆö a derived state. | in progress, active (ambiguous with trip status) |
| **Parking Lot** | The list of unplanned items within a phase. Not a separate data structure ŌĆö a filtered view of items where status = unplanned. Visible on day views as phase-scoped suggestion cards. Colloquially "ideas." | backlog, wishlist, ideas list |
| **Trip Goal** | *(v4 ŌĆö shipped: `/goals`, `/goals/capture`, `/goals/[id]`. The closeout-review step is **not yet wired** ŌĆö the wizard has no goals step as of 2026-06-12, #116 audit.)* A trip-level aspiration that isn't specific enough to be an item: "try paella," "do a wine tasting." Phase-less, location-less, time-less. A trip-scoped collection, sibling to phases (own sub-tab under Itinerary), captured with its creator. Zero or more items may be *linked* to a goal (a cross-cutting reference, not ownership ŌĆö the item still lives in its phase). **Status** mirrors Item Status and is **derived** from the highest-maturity linked item, ordered `done > planned > unplanned > considered` (so a goal is `considered` only when every linked item is); a goal with no linked items takes a **manual** status instead. Not reviewed at closeout ŌĆö the #195 goals-review step was **killed** in the 2026-06-15 grill ("nobody cares"); goals don't appear in the record. The *capture* half of the group-input cluster (see [[Swipe-Quiz]] for the *harvest* half). **Terminology:** the UI says "goal" everywhere ŌĆö "wish" is a deprecated synonym, never shown to users (#92). | bucket list item, achievement, ~~wish~~ (deprecated) |
| **Trip Member** | A user (or placeholder) associated with a trip. Has a role. | participant, collaborator |
| **Role** | Permission level: owner, co_owner, traveler, viewer. Governs what actions a member can take. | permission, access level |
| **Placeholder Member** | A trip member without a user account (`user` empty). Has a display_name and optional `placeholder_email`. [[Claim]]able when the real user signs up ŌĆö *unless* it is a [[Departed Member]] (tombstone), which is never claimable. **Surfaced to users as 'offline member'** (the '(offline)' badge on Members); 'placeholder' is internal-only (#116 audit). | ghost member, stub |
| **Departed Member** | *(#133 ŌĆö shipped: tombstone rows + "Former members" section on Members; **self-leave UI not built**.)* A [[Trip Member]] who has been removed (or self-left) but whose `trip_members` row is **retained as a tombstone** so the records they authored keep their identity ŌĆö above all [[Expense]]s and [[Settlement]]s, which can never lose their payer. Mechanically a [[Placeholder Member]] variant: `user` cleared (access severed), `name` snapshotted into `display_name`, `removed_at` set; rendered with a distinct **tombstone avatar**, not initials. Off the active roster; **never** [[Claim]]able and excluded from every join surface. The row hard-deletes only if nothing references it after removal. | removed member, ex-member, ghost |
| **Avatar** | A user's profile picture. A **user-level** identity asset stored once on the user (`users.avatar`) ŌĆö persists across every trip, unlike the per-trip display name on [[Trip Member]]. Shown wherever a member appears (vote stacks, members list, assignees, comments, notifications). [[Placeholder Member]]s have none and render initials; on [[Claim]] the claimer's avatar appears automatically. Visible only to co-travelers: **co-membership of a shared trip is the trust boundary** that lets one member read another's `users` row ŌĆö name and avatar only, never email or auth secrets. Falls back to a deterministic initials chip when absent. | profile pic, photo, gravatar |
| **Assignment** | A [[Trip Member]] **assigned to** an [[Item]] (`assigned_to`, many per item) is **doing / responsible for** it ŌĆö "I am doing this." General to all Item types, motivated by flights ("this is my flight"). **Self-assign** ("+ Me") is one tap and **immediate** ŌĆö it edits `assigned_to` on an existing item, *not* a [[Suggestion]], so it never enters review even when [[Auto-approve]] is off; open to **traveler / co_owner / owner**, never **[[Role\|viewer]]s**. Assigning *others* is the deliberate item-form path. Surfaced as **[[Avatar]]s on the item card** (ADR-0011), shown only when the trip has >1 member. *(planned ŌĆö #210, `docs/ITEM_ASSIGNMENT_PRD.md`.)* | assignee, responsible member, ~~paid_by~~/~~booked_by~~ (dead) |
| **Suggestion** | A proposed [[Item]] (or comment) submitted by a traveler, reviewed (pending ŌåÆ approved/rejected) unless [[Auto-approve]]d. *(#202 contribution-loop PRD: a pending suggestion is now visible to the whole group as a [[Ghost Card]] in its phase's [[Parking Lot]] and is votable ŌĆö review gates whether it becomes a **real** item, no longer whether the group can **see** it; rejection requires a note and archives it.)* | proposal, request |
| **Ghost Card** | *(planned ŌĆö #202.)* A pending [[Suggestion]] rendered in its [[Phase]]'s [[Parking Lot]] as a dotted-border placeholder, visible to all members and **votable** (the group signals support via [[Vote]]s before the owner decides). On approval it becomes a real [[Item]] ŌĆö carrying its votes over, credited to its author; on rejection (note required) it leaves the parking lot and is archived. Seen and heard immediately; approval only makes it *real*. | pending card, proposed idea |
| **Auto-approve** | Trip-level setting, on by default. When on, traveler contributions become unplanned items immediately. When off, contributions enter the suggestion review queue and the UI shows the traveler that their input will be reviewed by the trip owner. | |
| **Item Cost** | A single planning **estimate** on an Item (`cost_estimate_usd`, surfaced as "Cost"). Forward-looking ŌĆö "what we expect to pay." It is **not** a payment record: who actually paid, the real amount, and the split live on an [[Expense]] (linked back via `expenses.linked_item`). An Item never carries a payer ŌĆö "paid by" is an Expense concept. *(`cost_actual_usd` is deprecated and collapsed into the single Cost; `paid_by`/`booked_by` on Items are unused ghosts ŌĆö see `docs/CARD_CONTENT_SPEC.md`.)* | actual cost, paid_by, price |
| **Expense** | A money record tied to a trip: who paid, how much, how it splits. The authoritative payment/settlement record ŌĆö distinct from an [[Item Cost]] estimate. *(planned ŌĆö #211/#229: an Expense can be captured from an [[Item]] at the **[[Paid Moment]]** ŌĆö a persistent "Log payment" affordance, **not** a booked-fired toast ŌĆö turning prepaid plan spend into recorded money so the mid-trip "spent" is real. The 2026-06-19 re-grill (ADR-0014) **reversed** the original booked-flip trigger: paid-state is derived from the linked Expense, decoupled from [[Booking Readiness]].)* | cost, charge, payment |
| **Paid Moment** | *(planned ŌĆö #229; prefill mechanism #228; ADR-0014.)* The act of recording an [[Expense]] against an [[Item]] at the moment money leaves an account, surfaced as a persistent **"Log payment"** affordance on any payable item (every [[Item Type]] except `note`) that has **no linked [[Expense]] yet**. An item is **"paid" iff it has Ōēź1 linked [[Expense]]** (`expenses.linked_item`) ŌĆö there is **no stored paid flag**. Once linked, the affordance flips to a **"Paid $X"** summary (with a link-out to the item's expenses); a further payment (depositŌåÆbalance) stays loggable there but is no longer prompted ŌĆö **never nag, never block**. **Decoupled from [[Booking Readiness]]**: booking is a reservation, payment is money out (hold-now / charge-later / deposit-then-balance mean `booked` ŌēĀ paid). Capturing prefills the Expense from the item (amount ŌåÉ [[Item Cost]] estimate, editable & blank if none; description ŌåÉ title; payer ŌåÉ current member, editable; split ŌåÉ whole group, editable; date ŌåÉ today, backfilled if logging late). Open to any non-[[Role\|viewer]]; the shared linked-Expense state gives cross-member dedupe. | booked moment (deprecated), log-what-you-paid toast, paid flag |
| **Settlement** | A payment between two members to settle debts. | reimbursement |
| **Budget** | Per-category spending limits for a trip. Mode is per_day or total. *(planned ŌĆö #211: surfaced mid-trip in a Trip-Mode **Money** tab as a read-only glance ŌĆö spent vs budget with a **pre-trip / on-trip** split by date and what's **left**; "spent" is actual [[Expense]]s only, never [[Item Cost]] estimates.)* | |
| **Debt Simplification** | Algorithm that reduces N pairwise debts to a minimal set of transfers (greedy creditor/debtor matching). | |
| **Money Unit** | *(planned ŌĆö #230; ADR-0015.)* A self-declared, shared, leave-able grouping of trip members who pool money (a couple, or any ad-hoc pool); a solo member is a unit of one. **Settle-up collapses to the unit**: debts between unit members wash, and any one member can settle the unit's net debt to others ŌĆö splitting is unaffected (always per-person). The Money glance **auto-scopes to the viewer's own unit** (spent and budget summed across its members). A unit may set a **custom budget override** ŌĆö an absolute target, decoupled (it does not redistribute to other units or change the group total). | money circle, couple, household, money pool |
| **Vote** | A member's preference on a votable target. One vote per member per target. Four options (value id ŌåÆ surfaced label): Love (+2), Like (+1), Flexible (0), `dislike` (-2, surfaced as **"Pass"** per `VoteButtons.svelte` ŌĆö the id stays `dislike`). Score is never shown numerically ŌĆö only the relative order and who voted what (avatar stacks). *(On item **cards**, vote avatars are replaced by an **icon + count** pill ŌĆö the card avatar slot now shows [[Assignment\|assignees]] per ADR-0011; the who-voted-what avatar stacks remain on item **detail**.)* Two independent targets, each its own collection: **items** (`votes`) sort the parking lot; **goals** (`goal_votes`) sort the Goals/capture view. Votes never roll up across the goalŌåöitem link ŌĆö a goal's score is its goal-votes only, an item's its item-votes only (you can Love a goal yet Dislike a linked item). The shared scoring/stack logic lives in `voting.ts`; only the collection differs. | rating, star, like |
| **Swipe-Quiz** | *(v4 ŌĆö shipped: `/swipe/[phaseId]`, immersive, launched from the Phases sub-tab.)* A card-stack minigame for *harvesting* member preferences: walks one member through trip items, one card at a time, casting a [[Vote]] per card. The deck holds only items that member **hasn't voted on yet** (planned **and** unplanned); a voted card drops off, so the stack drains to empty. Scope is **phase-scoped** (v4 ŌĆö whole-trip sweep deferred); finishing one phase offers to continue to the next. Launches from a card on the **Phases sub-tab**. Writes the same `votes` records as the per-item vote buttons (one vote per member per item); re-voting happens on item detail, not in the deck. The *harvest* half of the group-input cluster; the *capture* half is the [[Trip Goal]] minigame. | swipe deck, tinder, quiz |
| **Document** | *(v4 ŌĆö shipped: S0ŌĆōS2 + offline precache; clipboard paste #73 and preview surfaces #72 pending.)* An **artifact** ŌĆö a file (PDF) or image ŌĆö uploaded to a trip. Has a **parent scope**: attaches to exactly one Item, or directly to the Trip ŌĆö no Phase scope. Item-scoped documents are execution-critical confirmations (boarding pass, hotel voucher, train ticket, tour confirmation); trip-scoped documents have no single item (travel insurance, master packing PDF). Artifacts only ŌĆö copy-pasteable text codes live on the Item's `confirmation_codes` field, **not** here. Added via file-picker upload **or clipboard paste** (e.g. a screenshot of a confirmation email). **Plain-stored, no encryption** ŌĆö protected by the same trip-membership rules that gate all trip data. One `documents` collection. Available offline in Trip Mode. | attachment, file |
| **Trip Documents** | The aggregate view of all Documents in a trip ŌĆö a read-only list unioning every item-attached and trip-attached Document. A *view* over Documents, not a copy: each artifact is one record surfaced in two places (its Item detail and the Trip Documents list). **Grouped by Item Type** (Lodging, Flights, Transportation, Activities, Meals, Notes) plus a **Trip-level** section for trip-scoped Documents. Occupies the nav slot formerly held by the **Vault**. | document folder, files tab |
| **Vault** | *(Retired in v4.)* Was client-side-encrypted storage for sensitive text (booking codes, passwords) ŌĆö AES-GCM + PBKDF2, lossy if password forgotten. **Encryption removed and the module renamed to [[Trip Documents]]**: a new app hasn't earned the trust to win a security contest against LastPass/1Password, and the marginal protection (operator/at-rest only ŌĆö trip members share the password) didn't justify the cost. Booking codes now live as plain text on the Item's `confirmation_codes` field; artifacts live in [[Trip Documents]]. | encrypted store, secure store | |
| **Memory** | *(v4 ŌĆö not yet built.)* One [[Trip Member]]'s captured recollection of one [[Day]]: an optional **photo** *and* an optional short **thought** (text) ŌĆö **at most one of each, per member, per day** (a hard cap, enforced by the data model ŌĆö a curated highlight, *not* a journal). Authored and editable only by its member; **visible to all trip members**; **never** surfaced in the [[Public Archive]] (memories are for the travelers, not the world). Created via the end-of-day **Note Before Bed** prompt, live during the day, or retroactively during [[Closeout]]. Distinct from a [[Document]]: a Document is a reference artifact used *during* the trip; a Memory is for remembering it *after*. | journal entry, photo, note, post, log |
| **Note Before Bed** | *(v4 ŌĆö not yet built.)* The end-of-day [[Trip Mode]] prompt (day-wrapped state) that invites each member to capture their [[Memory]] for the day. Optional and dismissable ŌĆö never nagging. A capture surface for the Memory, not a separate entity. | nightly journal, bedtime prompt |
| **Item Status** | Required field on every item. Lifecycle: unplanned (on the trip as an option, phase required, day optional) ŌåÆ planned (committed, phase and day required, may need booking) ŌåÆ done (closeout: we did it, day required) or considered (closeout: we didn't do it, day optional ŌĆö covers both skipped-planned and never-planned items). Colloquially, unplanned items are "ideas." | state |
| **Closeout** | End-of-trip review wizard. Walks through planned items day-by-day ŌĆö mark each done or considered. At the end of each phase, reviews unplanned items: keep for archive (marked considered) or remove, with an option to bulk auto-consider. New items can be added during closeout for things that happened spontaneously. *(#195, #116 audit: the per-phase **unplanned/ideas review** described above is specced but **not yet built** (WP-B-020) ŌĆö so ideas are currently dropped from the record. #195 (grilled 2026-06-15): travelers' closeout = the done/considered walk of planned items only; the unplanned/ideas review becomes an **optional, owner/co-owner-only** curation step (skipped by default) because those ideas are **public recommendations**, not group nostalgia (reverses WP-B-020's premise). **Goals-review is killed.** The final publish is a **binary** owner/co-owner control (also standalone on the closed trip ŌĆö see [[Public Archive]]). Gated to the wrap-up/closed lifecycle; opens to **travelers** for the item walk (SPEC ┬¦4 change).)* *(terminology, #116: the wizard's per-item verbs surface as **Done / Swap / Skip**; 'considered' shows only as a parenthetical on Swap, and Skip currently leaves the item planned ŌĆö done/considered is the data model, not the surfaced labels.)* | wrap-up, post-trip |
| **Public Archive** | Read-only, token-gated view of a closed trip. PII stripped. Published after configurable delay. The record = what we did + an **opt-in budget summary** (total / per-person, never itemized) + a **"what we considered"** section (curated Parking Lot ideas, as public recommendations).)* | shared link |
| **Notification** | In-app alert (no email/push). Types: suggestion_added, comment_added, member_joined *(#202 adds suggestion_approved + suggestion_rejected, sent to the suggestion's author).* | |
| **Invite** | A pending, **email-bound, single-use** invitation to one named person. Has a code, role, and 7-day expiry; accept requires the authed user's email to match the invite email. | **Join Link** | *(#118 ŌĆö shipped: `/join/[token]`, per-role create/copy/rotate/revoke on Members.)* A **shared, reusable** trip-level join token ŌĆö pasteable into a group text, no email targeting. ## Capabilities

> **Canonical model: `docs/CAPABILITY_MAP.md`** (full per-capability facets, maturity, cross-edges). This is the glossary-level summary. Replaces the former 7 "Bounded Contexts": the old **Collaboration** split into **Group Input** + **People & Membership**; **Trip Mode** ŌåÆ **Trip Execution**; **Shell** ŌåÆ **Platform**; and **Documents**, **Logistics**, **Ideation**, and **Integrations & Syncing** are now first-class.

**8 core capabilities:**

1. **Ideation** -- decide where/when/why to go (proposals, candidate scenarios, availability, goals). Mostly a frontier today.
2. **Itinerary** -- the shared plan, built and made ready: Trips, Phases, Days, Items (keystone), Parking Lot, Scheduling; **sub: Logistics** (Checklists, Tasks, readiness rollup, travel view; sub-sub: Booking).
3. **Group Input** -- converge the group: the Vote + Comment **mechanisms** (3 separate vote collections per ADR-0004/0009; Comment latent), the suggestion/contribution loop, swipe-quiz.
4. **People & Membership** -- right people, right access: Members + Roles (keystone), invites, join links, claims, tombstones; subs: Identity, Onboarding.
5. **Money** -- expenses, splits, settle-up, the per-person glance; sub: Budgeting.
6. **Documents** -- trip-private artifacts and confirmation codes (ADR-0016).
7. **Trip Execution** -- the live day-of lens (Now/Today, offline); sub: Light Replanning. Owns no native data.
8. **Records & Archive** -- the trip's legacy: Closeout, Public Archive, Portability; sub: Trip Memory.

**2 enabling capabilities:**

- **Platform** -- Auth (OTP), PWA/offline, Navigation, Design Tokens, Trip clock, Notifications. Infrastructure under the domain.
- **Integrations & Syncing** -- external connectors: Google Places + AeroDataBox enrichment, Resend email, map linkouts, calendar sync, photo-album linkout.

**Not a capability ŌĆö a data type:** **Saved Reference** (a public/general external link; the manual twin of enrichment; distinct from a trip-private Document). *(Vault retired ŌĆö ADR-0005.)*

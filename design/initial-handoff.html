00 · Integration philosophy

The goal: land design changes as theme + content swaps, not rewrites. Build the three things that don't change first — tokens, data model, mode state — then layer components on top. Screens are the most volatile layer and should come last.

Build order

  1. Tokens — lift §02 verbatim into src/theme/tokens.ts and CSS vars. These are stable.
  2. Schema + seed data — §03. Ship with the Andalucía trip as a seed so you can test with real shape from day 1.
  3. Mode state — §04. Put this in a root-level context; almost every screen reads it.
  4. Primitives — §05: Pill, Avatar, TypeIcon, Card, Button, NavBar, BottomNav, Phone. These have frozen APIs. Skin is locked.
  5. Screens — §06. Wire them to primitives + data. Design will iterate here; that's fine.
  6. Collaboration + sync last — §07.
Rule of thumb. If a primitive's API changes, every screen has to re-render mentally. So lock primitives early. If a screen's layout changes, only that screen does. So leave screens flexible.

03 · Data model

TypeScript discriminated unions. Item types drive both icon and the set of fields.

type Trip = {
  id: string;
  title: string;            // "Andalucía Loop"
  tagline?: string;         // "Nendaz, Lucerne, Zürich."
  startDate: ISODate;
  endDate: ISODate;
  status: 'draft' | 'upcoming' | 'on_trip' | 'past' | 'archived';
  cover?: Image;
  phases: Phase[];
  members: Member[];
  createdBy: UserId;
};

type Phase = {
  id: string;
  name: string;              // "Granada"
  startDate: ISODate;
  endDate: ISODate;
  days: Day[];
};

type Day = {
  id: string;
  date: ISODate;
  label?: 'travel_day' | 'rest_day';
  slots: Record<SlotKey, Item[]>;   // morning/afternoon/evening/anytime
};

type Item =
  | LodgingItem | TransportItem
  | ActivityItem | MealItem
  | NoteItem | ChecklistItem;

interface BaseItem {
  id: string;
  type: ItemType;            // drives icon family
  sub?: ItemSubtype;         // drives icon variant
  title: string;
  startsAt?: ISODateTime;   // when present, slot is inferred
  endsAt?: ISODateTime;
  durationMin?: number;
  location?: Location;
  status: ItemStatus;        // see §04
  bookedBy?: UserId;
  confirmationCodes: ConfCode[];
  cost?: Money;              // per-person or total + split
  attendees: UserId[];       // opt-in, defaults to all
  alternates: ItemId[];      // collapsed alts — live in parking lot
  parkingLot?: boolean;      // true = unscheduled idea
  proposedBy?: UserId;       // for suggestion inbox
  comments: Comment[];
}

type ItemType =
  'lodging' | 'transportation' | 'activity' | 'meal' | 'note' | 'checklist';

// Exhaustive sub-types drive icon set.
type ItemSubtype =
  // lodging
  'hotel' | 'airbnb' | 'hostel' | 'resort' | 'chalet'
  // transport
  | 'flight' | 'train' | 'bus' | 'car_rental' | 'ferry' | 'rideshare' | 'walk'
  // activity
  | 'tour' | 'museum' | 'outdoor' | 'sport' | 'shopping'
  | 'spa' | 'nightlife' | 'sightseeing'
  // meal
  | 'breakfast' | 'brunch' | 'lunch' | 'dinner' | 'drinks' | 'snack' | 'coffee';

type Member = {
  userId?: UserId;           // undefined → invited but not joined
  email?: string;
  displayName: string;
  initial: string;
  avatarUrl?: string;
  role: 'owner' | 'editor' | 'viewer';
  joinedAt?: ISODateTime;
};

type Suggestion = {
  id: string;
  kind: 'new_item' | 'change' | 'comment';
  proposedBy: UserId;
  targetItemId?: ItemId;
  payload: Partial<Item>;
  rationale?: string;
  createdAt: ISODateTime;
  state: 'pending' | 'approved' | 'auto_approved' | 'rejected';
};
Gotcha. A member with userId === undefined renders as a dashed placeholder avatar everywhere (Avatar placeholder). Don't filter them out in queries — the app shows them by design.

04 · Mode + state machines

Trip Moderoot state

Auto-engages, but manually overridable. Per-trip, not global.

type TripMode = 'planning' | 'trip';

// Auto-engage rule (runs on open + timer every 10min while open):
function defaultModeFor(trip: Trip, now: Date): TripMode {
  const startedButNotFinished =
    now >= trip.startDate.minus({ hours: 12 }) &&
    now <= trip.endDate.plus({ days: 1 });
  return startedButNotFinished ? 'trip' : 'planning';
}

// User override persists UNTIL next mode-boundary transition.
// i.e. if a user flips to Planning at 9am on day 2, they stay in Planning
// until the trip ends; a new auto-engage applies to the NEXT trip window.
UX contract. Editing is available in both modes. Trip mode simply reshapes the day view around now (what's current, what's next) and swaps the accent to clay. Don't gate features behind the toggle.

Item status

type ItemStatus =
  | 'idea'           // in parking lot, unscheduled
  | 'scheduled'      // on a day, not yet booked/confirmed
  | 'to_book'        // needs booking action
  | 'pending'        // booking submitted, awaiting confirm
  | 'booked'         // confirmed, has conf code
  | 'in_progress'    // now() within [startsAt, endsAt]
  | 'done'           // past endsAt
  | 'cancelled';

07 · API surface

Minimal set to light up the prototype. Real-time hooks come from your backend client.

// Trips
GET    /trips                         → Trip[]        // home page
POST   /trips                         → Trip
GET    /trips/:id                     → TripWithChildren
PATCH  /trips/:id                     → Trip

// Items — upsert across day/slot
POST   /trips/:id/items               → Item
PATCH  /items/:id                     → Item
DELETE /items/:id                     → 204
POST   /items/:id/move                → Item          // { dayId, slot }
POST   /items/:id/park                → Item          // move to parking lot

// Suggestions (collab)
POST   /trips/:id/suggestions         → Suggestion
POST   /suggestions/:id/approve       → Suggestion
POST   /suggestions/:id/reject        → Suggestion

// Realtime — WS channel per trip
ws://…/trip/:id   events:
  item.created | item.updated | item.moved
  suggestion.created | suggestion.resolved
  member.joined | comment.added

// Archive (public)
GET    /public/archive/:slug          → PublicTrip     // no auth, read-only

09 · Open questions

  1. Timezone handling: item startsAt — store UTC and convert at the phase's timezone, or store local + IANA zone? Decision blocks NOW divider accuracy.
  2. Cost splits: per-item split vs trip-wide ledger? Prototype shows per-item "$208 · split 4 ways" — need to model who-owes-who.
  3. Attendance model: default "all going" vs explicit opt-in per item? Prototype implies "all going unless removed".
  4. Offline: trip mode implies poor connectivity. Do conf codes + day view need full offline cache, or is "last synced" good enough?
  5. Photo album: Apple Photos integration is native-only. Web archive falls back to a gallery upload?
